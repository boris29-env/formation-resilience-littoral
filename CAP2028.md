# Cap 2028, atelier d'intelligence collective (IGEDD, mission 016714-01)

Application web autonome, un seul fichier (`cap2028.html`), pour animer en visioconférence un atelier de production d'idées avec 10 a 12 personnes. Aucune inscription pour les participants : un lien, un code de salle, un pseudonyme.

Six méthodes d'animation sont intégrées, pilotées en direct par l'animateur :

1. Design fiction, écrire des scénarios prospectifs de sortie.
2. Codéveloppement, une situation exposée, le groupe conseille (cinq étapes).
3. Six chapeaux, penser le sujet sous six angles successifs.
4. Débat mouvant, se positionner sur une affirmation puis argumenter.
5. World café, trois tables thématiques et contributions croisées.
6. Cartographie d'arguments, structurer le pour et le contre d'une thèse.

## 1. Tester immédiatement, sans rien configurer

Ouvrez `cap2028.html` dans Chrome ou Edge. Sans configuration Firebase, l'application fonctionne en mode local : la synchronisation marche entre plusieurs onglets d'un meme navigateur, ce qui suffit pour découvrir l'outil seul (ouvrez deux onglets sur la meme salle, l'un en animation).

## 2. Mots de passe d'accès

Deux niveaux d'accès sont définis en haut du script de `cap2028.html`, dans le bloc `ACCESS` :

```js
const ACCESS = {
  participant: "igedd2028",   // exigé pour entrer dans l'atelier
  animateur:   "anim2028"     // exigé pour ouvrir le panneau d'animation
};
```

- Le **mot de passe participant** s'affiche sur l'écran d'accueil ; tout le monde le saisit pour entrer.
- Le **mot de passe animateur**, distinct, est demandé au premier clic sur le bouton « Animation » et débloque le pilotage (choix de la méthode, bibliothèques, export).

Changez ces deux valeurs avant chaque atelier et communiquez-les par le canal de la visioconférence. Laissez `""` pour désactiver un niveau. La vérification se fait côté navigateur : pour un atelier fermé et temporaire, cela écarte les visiteurs de passage, mais le code source restant lisible, ce n'est pas un secret inviolable. Pour un usage sensible, ajoutez le contrôle d'accès côté base (règles Firebase, code de salle non devinable).

## 3. Brancher Firebase pour une session multi-postes

1. Créez un projet sur https://console.firebase.google.com
2. Activez **Realtime Database** (choisissez une région européenne, par exemple `europe-west1`), en mode test pour commencer.
3. Dans `cap2028.html`, repérez le bloc `FIREBASE_CONFIG` en haut du script et collez votre configuration (clés fournies par Firebase, rubrique « Paramètres du projet », « Vos applications », application Web).
4. Régles Realtime Database conseillées pour un atelier fermé et temporaire :

```json
{
  "rules": {
    "rooms": {
      "$room": { ".read": true, ".write": true }
    },
    "library": { ".read": true, ".write": true }
  }
}
```

Le nœud `library` héberge les situations de codéveloppement ajoutées par les animateurs depuis le panneau d'animation : ces ajouts sont conservés entre les ateliers et viennent compléter la bibliothèque livrée dans `content.js`.

À durcir ensuite (code de salle non devinable, expiration). Voir avec votre DSI pour la conformité RGPD ; Firebase étant un service hébergé hors UE, une base auto-hébergée (Supabase, ou base souveraine) sera préférable pour un usage pérenne sur données RH.

## 4. Héberger sur GitHub Pages

1. Créez un dépôt, déposez-y `cap2028.html` et `content.js` (et ce `README.md`). Les deux fichiers doivent rester dans le même dossier : `content.js` contient les bibliothèques de contenu importées par l'application.
2. Dans `Settings` puis `Pages`, publiez la branche principale.
3. Partagez aux participants l'URL avec le parametre de salle, par exemple :
   `https://votrecompte.github.io/votredepot/cap2028.html?salle=igedd-2028`

L'animateur ouvre le bouton « Animation » en haut a droite pour choisir la méthode active et piloter chaque séquence.

## 5. Bibliothèques de contenu (`content.js`)

Le panneau d'animation propose, pour chaque méthode, un sélecteur « Bibliothèque » alimenté par `content.js` :

- 12 questions prospectives et 16 signaux faibles (design fiction) ;
- 6 situations anonymes proposables comme cas fictif (codéveloppement), enrichissables par l'animateur depuis le panneau d'animation (les ajouts sont partagés entre ateliers, stockés sous `library/codev/situations`) ;
- 4 sujets avec une consigne par chapeau (six chapeaux) ;
- 13 affirmations clivantes (débat mouvant) ;
- 4 jeux de trois tables (world café) ;
- 6 thèses avec amorces pour et contre (cartographie), avec un bouton « Pré-amorcer » qui dépose les amorces dans l'arbre pour éviter la page blanche.

Chaque item est étiqueté avec ses axes thématiques (A à H) de la mission, affichés à l'animateur pour équilibrer la séance. Le contenu respecte le cahier de contenu de la mission : registre institutionnel, anonymat strict, énoncés défendables des deux côtés, aucun chiffre inventé.

## 6. Récupérer le matériau

Le bouton « Exporter (JSON) » du panneau d'animation télécharge l'ensemble des contributions (idées, scénarios, arguments, positions) dans un fichier structuré, prêt à archiver dans le dossier de la mission.
